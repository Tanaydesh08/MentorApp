What is working now
- Users can register as `STUDENT` or `MENTOR`.
- Login returns a JWT token and the frontend stores it in `localStorage`.
- Authenticated users can open `/dashboard` and call `GET /api/users`.
- Dashboard can create a new session id or join an existing session route.
- `/session/[sessionId]` supports realtime presence, chat, shared Monaco editor sync, and browser media controls.
- WebRTC signaling is relayed through the Spring WebSocket server.

Current architecture in one sentence
- Frontend owns the pages, JWT storage, session UI, Monaco editor, browser media, and WebRTC peer logic; backend owns auth, JWT validation, the users API, websocket handshake validation, and room message relay; Oracle currently stores only users.

What is only partially complete
- Session ids are generated on the frontend and are not stored in the database.
- Chat messages and editor state are not persisted.
- WebRTC is tuned for a 1-on-1 demo flow and still depends on browser permissions, STUN, and network quality.
- There is no TURN server, no refresh-token flow, and no production deployment setup.
- Automated testing is still minimal.

Highest-value next steps
1. Add `sessions`, `messages`, and `session_invitations` tables/entities.
2. Move session creation and join validation into backend APIs.
3. Add TURN server support and stronger reconnect/recovery handling.
4. Add migration tooling plus frontend/backend tests.
5. Prepare production configuration and deployment.
