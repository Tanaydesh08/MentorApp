Current frontend implementation
1. App shell and styling
- `layout.tsx` provides the root HTML/body shell.
- `globals.css` defines the visual theme, colors, and shared CSS variables.
- `page.tsx` is the landing page that routes users to register or login.

2. Auth UI and token usage
- `register/page.tsx` sends email, password, and role to the backend.
- `login/page.tsx` sends credentials and stores the returned JWT in `localStorage`.
- `src/lib/auth.ts` decodes token payloads, extracts email, clears the token, and generates 6-character session ids.
- The helper can also read role claims if a JWT contains them, but the current backend token only stores the email subject.

3. Backend API integration
- `src/lib/api.ts` centralizes `registerUser`, `loginUser`, and `getUsers`.
- The REST base URL is driven by `NEXT_PUBLIC_API_BASE_URL` and falls back to `http://localhost:8080`.
- `getUsers` sends the stored JWT as a Bearer token.

4. Dashboard flow
- `dashboard/page.tsx` protects the page by checking for a stored token.
- It fetches `/api/users` after login and renders the returned users.
- It can create a fresh session id or route into an existing `/session/[sessionId]`.
- The dashboard reads the current email from the JWT; current role display is only available if the JWT includes role data.

5. Session room flow
- `session/[sessionId]/page.tsx` is the main realtime workspace.
- It opens a websocket connection to `/ws/session` with `sessionId`, `token`, and a per-tab `clientId`.
- The page shows local video, remote video, a shared editor, and realtime chat in a single layout.

6. Chat
- Chat messages are sent through the websocket as room events.
- The local sender message is rendered immediately in the UI.
- Same-client websocket echoes are ignored so messages do not duplicate.

7. Shared editor
- Monaco Editor is integrated with JavaScript and Python options.
- Editor updates are debounced and broadcast through the websocket as `editor` messages.
- Sync is last-write-wins; there is no CRDT/OT conflict-resolution layer yet.

8. WebRTC and media
- `startMedia()` requests camera and microphone access with `getUserMedia`.
- `ensurePeerConnection()` creates the `RTCPeerConnection` and wires ICE, track, and negotiation handlers.
- `handleSignalPayload()` applies offers, answers, and ICE candidates received through websocket messages.
- `toggleTrack()` and `toggleScreenShare()` manage mic, camera, and screen-sharing behavior.
- Current ICE config only uses a public STUN server.

9. Presence and signaling
- Presence messages drive the participant list and join/leave system messages.
- A per-tab `clientId` is used so multiple tabs from the same email do not confuse signaling logic.
- The page logic is designed for 1-on-1 sessions, not a production multi-peer room.

Current frontend limits
- No refresh token flow or Next.js middleware-based auth protection.
- No persistent session API; session ids are client-generated only.
- No stored message history or saved editor snapshots.
- No device selector, reconnect UI, or advanced failure recovery.
- No automated frontend tests in the repo.

Frontend maturity summary
- The frontend is demo-ready for the core mentor/student flow.
- The remaining work is mainly persistence, hardening, and production polish rather than missing first-pass screens.
