Key scope inferred from PDFs
The intended full product is a web-based 1-on-1 mentorship platform where mentor and student can:
- join a private session
- video call in real time
- chat via messages
- collaboratively edit code in a shared editor

Planned stack from the roadmap PDF
- Frontend: Next.js + TypeScript + Tailwind CSS + Monaco Editor + WebSocket client
- Backend: Spring Boot + Spring Security + WebSocket + JPA + PostgreSQL
- Auth: JWT-based auth
- Video: WebRTC

Current backend assumptions wired into frontend
Backend base URL currently hardcoded in:
- `src/lib/api.ts`

Current auth/backend endpoints used
- `POST http://localhost:8080/api/users/login`
- `POST http://localhost:8080/api/users/register`
- `GET  http://localhost:8080/api/users`

Important backend behavior confirmed from Spring Boot code
1. Login returns a JWT as plain text.
2. Register returns a JSON user response.
3. `GET /api/users` requires Bearer token authentication.
4. Spring Security currently restricts `GET /api/users` to users with authority `MENTOR`.

Practical consequence
- If a mentor logs in, the dashboard should show the users list.
- If a student logs in, the dashboard will show the unauthorized/error state unless backend security is changed.
